## 1. 产品概述

### 1.1 产品背景
原项目（[aqtw-random-equipment](https://github.com/HB-cat/aqtw-random-equipment)）是一个基于静态HTML的《暗区突围无限》游戏装备随机生成工具。因代码耦合度高、数据文件管理混乱、难以动态更新，现决定使用现代化技术栈进行规范化复写。

### 1.2 产品目标
*   **工程化**：用Python后端 + 现代前端框架重构，实现前后端分离、模块化、可测试。
*   **动态化**：装备数据不再写死在代码中，而是通过外部API动态拉取，模拟游戏服务器实时更新装备的效果。
*   **规范化**：建立清晰的项目结构、数据模型、冲突规则引擎，后续可维护和扩展。
*   **AI协作验证**：本项目将由AI编程工具（Trae）辅助完成编码，验证“人机协作独立交付”的可行性与效率，积累AI辅助开发的最佳实践。

### 1.3 目标用户
*   《暗区突围无限》玩家，希望随机生成符合战备值区间的合法装备搭配。
*   对装备搭配规则不熟悉的新手玩家。
*   追求趣味性或挑战性的硬核玩家。

## 2. 产品功能

### 2.1 核心功能：智能装备随机生成
*   **模式选择**：提供三种生成模式，对应不同战备值区间。
    *   **普通模式**：总价 < 450,000
    *   **封锁模式**：150,000 ≤ 总价 ≤ 1,000,000
    *   **强化封锁模式**：总价 ≥ 450,000
*   **装备类别控制**：用户可自由勾选需要生成的装备部位。
    *   **防弹衣/弹挂**：互斥选择（有甲弹挂 或 护甲+无甲弹挂）
    *   **头盔**：含普通头盔和自闭头
    *   **面罩**：需与所选头盔兼容
    *   **背包**、**耳机**、**防毒面具**、**枪械**（主武器+副武器）
*   **随机/指定模式**：每个部位可独立设置为“随机”或由用户“指定”某件装备。
*   **一键生成**：基于所选模式和类别，随机组合一套无冲突的装备，并计算总战备值。

### 2.2 核心规则引擎：装备冲突检测
程序需严格遵守以下规则，生成后整体校验，冲突则重试（最多50次）。
1.  **防毒面具冲突**：不能与任何面罩同时装备；不能与特定头盔（56K型直升机头盔、DOD9型防爆头盔）同时装备。
2.  **耳机冲突**：自闭头不能搭配耳机。
3.  **面罩兼容性**：面罩只能搭配其 `compatible_helmets` 列表中指定的头盔。
4.  **防弹衣/弹挂互斥**：若选择了有甲弹挂，则不能装备其他护甲或无甲弹挂。
5.  **重试机制**：最多尝试50次生成无冲突组合。失败时向用户反馈“生成失败，请调整装备类别开关或稍后再试”，并清空当前显示结果。

### 2.3 动态数据支持
*   **数据源**：所有装备数据（头盔、护甲、枪械等）由独立的模拟游戏服务器API提供。
*   **动态更新**：后端启动时从外部API拉取全量数据并缓存。当外部API数据变更时，后端可重新拉取，实现装备属性、价格的实时模拟更新。

### 2.4 用户界面（新前端）
*   **视觉风格**：沿用原项目的军事科技感设计，参考暗区突围官网。
    *   深色背景渐变（#0a0f1e）
    *   金属质感卡片
    *   发光按钮
    *   装备卡片带悬停效果
*   **交互体验**：
    *   响应式布局，支持桌面和移动端。
    *   装备卡片滑动淡入动画。
    *   冲突或失败时给予明确的文字提示。

## 3. 技术架构与选型

本次复写目标为“Python后端 + 新前端”的分离式架构。

| 模块 | 技术栈 | 选型理由 |
| :--- | :--- | :--- |
| **新前端** | **Vue 3 + Vite** | 组件化开发，生态好，适合AI辅助生成 |
| **后端** | **Python 3.10+** with **FastAPI** | 异步支持好，自动生成API文档，便于测试和对接 |
| **外部模拟数据库** | **独立 FastAPI 服务** | 独立进程，通过REST API提供装备数据，模拟游戏服务器 |
| **后端数据获取** | **httpx** (异步HTTP客户端) | 用于后端从模拟数据库异步拉取数据 |
| **数据缓存** | Python 内置 `dict` 或 `cachetools` | 在内存中缓存从外部API拉取的装备数据 |

### 项目结构（目标）
```
aqtw-equipment-generator/
├── backend/                # Python后端
├── frontend/               # Vue前端
├── mock-game-server/       # 模拟游戏数据API服务
├── CLAUDE.md               # 项目开发日志
└── README.md
```

## 4. 数据模型与接口定义

### 4.1 数据模型（后端及外部API共用）
```python
# 头盔
{
  "name": "SH40 头盔",
  "type": "头盔",         # "头盔" 或 "自闭头"
  "level": 4,
  "price": 12000,
  "mask_compatible": ["SH40面罩"]  # 可搭配的面罩列表
}
# 其他装备（面罩、护甲、弹挂等）参照原项目数据结构，字段完整。
```

### 4.2 外部模拟API设计
服务地址：`http://localhost:8001` （示例）
*   **获取全量装备**：`GET /api/equipment`
    *   返回所有部位（头盔、面罩、护甲...）的列表
*   **获取全量枪械**：`GET /api/weapons`
    *   返回所有类别（突击步枪、冲锋枪...）的枪械列表
*   **（可选）更新装备价格**：`POST /api/equipment/update` —— 用于演示动态数据

### 4.3 后端内部API设计
服务地址：`http://localhost:8000` （示例）
*   **生成装备组合**：`POST /api/generate`
    *   请求体 (Request Body)：
        ```json
        {
          "mode": "normal", // "normal" | "lockdown" | "hardcore"
          "categories": ["helmet", "armor", "weapon", ...], // 需要生成的部位
          "fixed_equipment": { "helmet": "SH40 头盔", ... } // 用户指定的装备（可选）
        }
        ```
    *   响应体 (Response Body)：
        ```json
        {
          "success": true,
          "loadout": {
            "helmet": {...}, "face_shield": {...}, ...
          },
          "total_price": 235000
        }
        ```
        或失败时：`{ "success": false, "message": "生成失败，请调整装备类别开关或稍后再试" }`

## 5. 开发实施计划（供 AI 协作参考）

此计划将分阶段交由Trae执行，每个阶段产出可验收的结果。

### 阶段一：基础设施与模拟环境
1.  根据上述项目结构，创建 `backend/`, `frontend/`, `mock-game-server/` 目录骨架。
2.  **编写数据转换脚本**：将原项目的 `.txt` 数据文件（Python字典格式）解析为标准的 `JSON` 文件。
3.  **搭建模拟游戏服务器** (`mock-game-server`)：创建一个简单的FastAPI应用，读取上一步的JSON文件，对外暴露 `GET /api/equipment` 和 `GET /api/weapons` 接口。

### 阶段二：后端核心逻辑开发 (`backend`)
1.  **实现数据提供层**：编写模块，通过 `httpx` 从模拟服务器异步获取并缓存装备数据。
2.  **实现规则引擎**：将4条核心冲突规则编码为 `check_conflicts(loadout)` 函数。
3.  **实现生成器**：编写核心随机生成逻辑，包含50次重试机制。
4.  **实现API路由**：创建 `POST /api/generate` 端点，串联上述模块。
5.  **编写单元测试**：重点测试每条冲突规则和生成器的边界情况。

### 阶段三：新前端开发 (`frontend`)
1.  使用Vue 3 + Vite初始化项目。
2.  实现视觉风格（军事科技感、深色主题、金属卡片）。
3.  实现核心交互：模式选择、装备开关、随机/指定、调用后端API。
4.  处理响应：展示生成的装备组合、总价，或显示错误提示。
5.  添加动画效果。

### 阶段四：联调与动态演示
1.  启动三个进程（模拟服务器、后端、前端），完成端到端联调。
2.  **动态数据演示**：手动修改模拟服务器的装备数据文件（或通过POST接口），刷新前端页面，验证重新生成的装备组合是否反映数据变化，形成“实时更新”的模拟效果。

1.  **功能完整**：三种模式、所有装备类别、随机/指定、冲突检测与重试机制均按预期工作。

3.  **动态演示**：修改模拟数据库中的数据（如装备价格），能在后续生成中看到变化。
4.  **视觉还原**：界面风格贴合暗区突围主题，交互流畅。
5.  **代码质量**：后端关键逻辑有单元测试覆盖，项目结构清晰，有完整的README。

## 7. 风险与应对

| 风险 | 可能性 | 应对措施 |
| :--- | :--- | :--- |
| AI生成的冲突检测逻辑有误 | 中 | 人工审查代码，重点测试边界情况（如自闭头+耳机+面罩的组合）。 |
| 外部API数据格式与实际不符 | 低 | 严格遵循本文档第4节定义的数据模型。 |
| 前端视觉风格还原度不达标 | 高 | 提供原项目截图和官方风格参考链接，多轮迭代调整CSS。 |
| 50次重试机制在某些条件下频繁失败 | 低 | 保留原项目重试逻辑，并可通过调整用户选择的类别范围来规避。 |
